üéØ Role Purpose & Context

Why this role exists and how it supports the CaniCol model.

Mission Statement

The Operations Integrator serves as the "mini-COO" of DriveOn Commerce, responsible for maintaining operating system coherence across all four pods. This role ensures that the CaniCol v4.0 model runs as designed, that canon remains the single source of truth, and that cross-pod friction is eliminated systematically rather than through ad-hoc heroics.

Strategic Context

DriveOn Commerce operates a lean-volume dropshipping engine testing 45-60 products/month with strict economic constraints ($80-$100 average P1+P2 spend). At this volume, system coherence is non-negotiable:

  • Without WIP discipline: Pods overflow, quality drops, kill discipline fails
  • Without canon integrity: Each pod invents its own rules, creating drift and inconsistency
  • Without bottleneck resolution: The 45-60 test/month engine stalls
  • Without cross-pod orchestration: Handoffs fail, products get stuck between gates
‚öñÔ∏è Canon Constraint

This role exists to implement canon, not modify it. Any proposed changes to L0-L2 economics, gate definitions, WIP caps, or pod structures must flow through formal Canon Change Requests (CCR) and founder approval. The Integrator protects the CaniCol model above all else.

How This Role Supports CaniCol Economics

  • Maintains test bank integrity: Ensures $4,000-$5,500/month P1+P2 spend stays within bounds
  • Enforces kill discipline: Tracks that 70-80% of products are eliminated in P1 within 24-48 hours
  • Protects winner rates: Validates that 4-6% hit rate expectations are realistic given current pod capacity
  • Prevents capacity bottlenecks: Ensures research, creative, and media pods can sustain 45-60 test/month throughput
  • Guards WIP caps: Stops pods from overloading (e.g., R1 ‚â§10, R2 ‚â§6, R3 ‚â§3 per FTE)

Workflow 1: Escalation Resolution & Bottleneck Clearing

Trigger

Pod lead or team member escalates an issue via Discord #p4-ops-escalations or ClickUp ops ticket.

Inputs

  • Escalation ticket in ClickUp (Space: "Pod 4 - Ops", List: "Escalations")
  • Context: Which pod(s), which product(s), what's blocked
  • Canon reference: Which L1-L3 doc is relevant (if known)

Steps

  1. Acknowledge within 2 hours: Post in Discord confirming receipt and expected resolution time
  2. Classify the escalation:
    • Type A - Process Clarification: Canon unclear, SOP ambiguous
    • Type B - Tool/Infra Issue: ClickUp/Notion/Discord not working as expected
    • Type C - Cross-Pod Conflict: Two pods disagree on handoff or responsibility
    • Type D - Capacity/WIP Violation: Pod is overloaded beyond canonical caps
    • Type E - Canon Contradiction: Reality doesn't match L1-L3 rules
  3. Check canon precedence:
    • Open Notion Canon Registry and locate relevant L0-L3 docs
    • Verify current version and status (ACTIVE vs LEGACY)
    • Apply precedence rule: L0 > L1 > L2 > L3 > L4
  4. Make decision based on type:
    • Type A: Clarify canon, document in decision log, flag SOP for update
    • Type B: Log in Change_Registry, coordinate fix with PM/VA
    • Type C: Reference L2-02 Pod Overview for RACI, make binding call
    • Type D: Review WIP data, coordinate capacity adjustment with Founder
    • Type E: Draft Canon Change Request, escalate to Founder
  5. Document decision:
    • Update ClickUp ticket with resolution and rationale
    • Add entry to Notion decision log with date, decision, canon refs
    • If canon/SOP update needed, create follow-up task
  6. Communicate resolution:
    • Post resolution in Discord escalation thread
    • Notify affected pod leads via @mention
    • If system-wide impact, announce in #sys-all-hands
  7. Close or hand off:
    • If resolved: move ClickUp ticket to "Closed"
    • If requires follow-up: assign to PM/VA and set due date
    • If requires Founder approval: move to "Awaiting Approval" and notify

Outputs

  • Resolved escalation ticket in ClickUp
  • Decision log entry in Notion
  • Discord resolution announcement
  • SOP update task (if needed)
  • Canon Change Request draft (if Type E)

Quality Check

  • ‚úì Decision aligns with canon precedence
  • ‚úì All affected pods notified
  • ‚úì Documentation is searchable and linked
  • ‚úì Follow-up actions assigned with clear owners

Time Budget

15-45 minutes per escalation depending on type complexity

Workflow 2: Canon Maintenance & Version Management

Trigger

Quarterly review cycle, or immediate trigger if L1-L2 change approved by Founder.

Inputs

  • Approved Canon Change Request from Notion Change_Registry
  • Current canon docs from Google Drive + Notion
  • Impact assessment (which L3/L4 docs affected)

Steps

  1. Open Change_Registry entry:
    • Navigate to Notion ‚Üí DriveOn ¬∑ Canon & SOPs ‚Üí 01 ¬∑ Canon_Config (SSOT) ‚Üí Change_Registry
    • Filter for Status = "Approved" or "In Progress"
    • Review the change scope, affected docs, and systems
  2. Update L0-L3 canon docs:
    • Open the source doc in Google Drive (e.g., L1-01_CaniCol_Model_v4.0.docx)
    • Make the approved changes following exact CCR specifications
    • Bump version number per L1-02 rules (MAJOR.MINOR.PATCH)
    • Update change log section with date, change description, author
    • Save new version with updated filename (e.g., v4.1 becomes v4.2)
  3. Update Canon Registry in Notion:
    • Open the affected doc's row in Canon Registry
    • Update Version field (e.g., 4.0 ‚Üí 4.1)
    • Update Status (confirm ACTIVE if previously DRAFT)
    • Update Last Reviewed date to today
    • Add Change Summary text
    • Set Replaces_Canon_Doc_ID to previous version
  4. Mark previous version as LEGACY:
    • Find the previous version row in Canon Registry
    • Change Status from ACTIVE to LEGACY
    • Add note: "Superseded by [new version]"
  5. Assess downstream impact:
    • Check which L3 infra docs reference the changed L1-L2 doc
    • Check which L4 SOPs embed the old values/rules
    • Create a list of docs requiring regeneration
  6. Trigger L3/L4 updates:
    • For each affected L3 doc: draft updated sections, bump version
    • For each affected L4 SOP: coordinate with LLM SOP factory to regenerate
    • Update SOP Library in Notion with new versions
    • Deprecate old SOP versions (status = LEGACY)
  7. Update tools to match canon:
    • If ClickUp structure changed: update spaces/folders/lists/statuses
    • If Notion schema changed: update DB properties
    • If Discord channels changed: rename or create channels
    • Document all tool changes in Change_Registry
  8. Communicate the change:
    • Draft change announcement for Discord #sys-canon-updates
    • Include: what changed, why, impact on operations, where to find new docs
    • @mention all pod leads
    • Post link to updated Canon Registry entry
  9. Update Change_Registry entry:
    • Move Status to "Implemented"
    • Link all updated canon docs in Docs_Links relation
    • Add implementation notes
    • Set review date for follow-up check (usually +2 weeks)
  10. Monitor adoption (2-week follow-up):
    • Check if pods are following new canon
    • Address any confusion or friction
    • Move Change_Registry status to "Reviewed" then "Closed"

Outputs

  • Updated L0-L3 canon docs with new version numbers
  • Updated Canon Registry entries (new ACTIVE, old LEGACY)
  • Regenerated L3/L4 docs reflecting new canon
  • Updated tool configurations (ClickUp/Notion/Discord)
  • Change announcement in Discord
  • Closed Change_Registry entry

Quality Check

  • ‚úì All version numbers incremented correctly per L1-02 rules
  • ‚úì Canon Registry shows clear ACTIVE vs LEGACY status
  • ‚úì Downstream L3/L4 docs updated and cross-referenced
  • ‚úì Tool configurations match new canon (no drift)
  • ‚úì All pods notified and aware of changes

Time Budget

2-8 hours depending on scope (PATCH = 2h, MINOR = 4h, MAJOR = 8h+)

Workflow 3: Infrastructure Change Management

Trigger

Proposed infra change from any pod, or proactive improvement identified by Integrator.

Inputs

  • Change proposal (can be informal request or formal CCR)
  • System affected: ClickUp, Notion, Discord, Drive, or Automation
  • Business justification: What problem does this solve?

Steps

  1. Create Change_Registry entry:
    • Open Notion Change_Registry DB
    • Create new entry with Title: "[Infra] [System] - [Description]"
    • Set Type = "Infra" or "Mixed"
    • Set Risk = "Low" / "Medium" / "High" based on impact
    • Document Requested_By, Systems_Affected
    • Status = "Proposed"
  2. Check canon alignment (Step 0 from L3-06):
    • Does this touch L1-01 CaniCol Model economics? (STOP if yes)
    • Does this touch L2-02 Pod Overview responsibilities? (Needs L2 update)
    • Does this touch L2-03 Role Registry or WIP caps? (Needs L2 update)
    • Does this touch L3-01 to L3-05 blueprints? (Needs L3 update)
    • If canon conflict detected: MUST update canon first before proceeding
  3. Classify change magnitude:
    • Minor: Cosmetic, view changes, no ID/flow impact ‚Üí Ops Lead can approve
    • Medium: New fields/statuses, new DBs/channels, no breaking changes ‚Üí Ops Lead + Founder approve
    • Major: Breaking changes to IDs, pod logic, flows ‚Üí Requires L2/L3 canon update + Founder approval
  4. If Medium/Major and touches canon:
    • Draft canon update (relevant L2/L3 doc)
    • Bump version per L1-02 rules
    • Update Canon Registry
    • Get Founder approval on canon change BEFORE implementing in tools
  5. Update Workspace_Map in Notion:
    • For new entities: add rows with Token, System, Human_Name, URL (placeholder), Used_By_Roles
    • For renamed entities: update existing row, mark old Token as Deprecated
    • For deleted entities: set Status = Deprecated (don't delete row)
  6. Implement in tools (Step 5 from L3-06):
    • ClickUp: Create/rename spaces, folders, lists, statuses, custom fields per Workspace_Map
    • Notion: Create/update DBs, properties, views per schema
    • Discord: Create/rename channels, categories per structure
    • Drive: Create/rename folders per hierarchy
    • Update Workspace_Map with actual URLs once created
  7. Update or create automations if needed:
    • Add row to Automation_Registry if new automation
    • Update existing row if automation modified
    • Document: Trigger, Input, Output, Owner_Role, Touches_Workspace_Tokens
    • Test automation in sandbox before enabling
  8. Communicate and train (Step 6 from L3-06):
    • Post announcement in Discord #sys-infra-changes
    • Include: what changed, why, where updated docs live
    • If workflow changes: record quick Loom walkthrough
    • @mention affected pod leads
  9. Regenerate affected SOPs (Step 7 from L3-06):
    • Identify SOPs that reference old structure
    • Coordinate with LLM SOP factory to regenerate with new infra references
    • Update SOP Library in Notion
    • Mark old versions as LEGACY
  10. Update Change_Registry to "Implemented":
    • Link updated canon docs if any
    • Link updated Workspace_Map rows
    • Link updated Automation_Registry rows
    • Set review date (+1 week for monitoring)
  11. Monitor adoption and close:
    • Check if pods are using new structure correctly
    • Address any friction or confusion
    • Move Change_Registry to "Reviewed" then "Closed"

Outputs

  • Updated infrastructure in live tools (ClickUp/Notion/Discord/Drive)
  • Updated Workspace_Map in Notion
  • Updated Automation_Registry if applicable
  • Updated canon docs if Medium/Major change
  • Regenerated SOPs reflecting new infra
  • Change announcement and training material
  • Closed Change_Registry entry

Quality Check

  • ‚úì Canon updated BEFORE tools (if needed)
  • ‚úì Workspace_Map accurately reflects live tool state
  • ‚úì No "shadow" entities exist outside Workspace_Map
  • ‚úì All automations documented in Automation_Registry
  • ‚úì SOPs regenerated and referencing correct structure
  • ‚úì Pods trained and using new structure correctly

Time Budget

1-6 hours depending on change class (Minor = 1h, Medium = 3h, Major = 6h+)

Workflow 4: System Health Monitoring & Bottleneck Detection

Trigger

Daily morning routine, or triggered by alert from BI dashboard or Discord notification.

Inputs

  • ClickUp WIP dashboards for each pod
  • BI portfolio dashboard showing ROAS, test bank burn, winner rate
  • Discord alerts from #sys-health channel
  • Notion Product OS status distribution

Steps

  1. Check overnight Discord alerts:
    • Open Discord #sys-health channel
    • Review any automated alerts or manual flags from pods
    • Note: WIP violations, budget overruns, stalled products, critical escalations
  2. Scan ClickUp WIP dashboards by pod:
    • Pod 1 (Research):
      • R1 WIP per researcher (should be ‚â§10)
      • R2 WIP per researcher (should be ‚â§6)
      • R3 WIP per researcher (should be ‚â§3)
      • R3 approval backlog (should be <5 days old)
    • Pod 2 (Creative):
      • Products in creative queue (should match R3 output)
      • WIP per creative role (4-6 products)
      • Time since R3 handoff for oldest item (should be <3 days)
    • Pod 3 (Media & Store):
      • P1 WIP per media buyer (should be 8-10)
      • P2 WIP per media buyer (should be 4-6)
      • Store setup backlog for new P1 tests
      • Kill discipline: products in P1_FAIL status >48h (should be 0)
    • Pod 4 (Ops):
      • Open escalations >48h (should be 0)
      • Active infrastructure initiatives (should be 8-10)
      • Pending Canon Change Requests (should have response within 24h)
  3. Review BI portfolio dashboard:
    • Test Bank Burn Rate: Current month P1+P2 spend vs. $4,000-$5,500 budget
    • Portfolio ROAS: Blended ROAS across all active products
    • Winner Rate: P2_PASS products as % of tested (should be 4-6%)
    • Kill Discipline: % of clear failures killed within 24-48h
    • Testing Velocity: Products entering P1 per week (should be 10-15)
  4. Check Notion Product OS status distribution:
    • Filter by Status: count products in each stage
    • Compare to canonical distribution: 70-80% kill in P1, 10-16% reach P2, 4-10% reach P3
    • Flag any anomalies (e.g., P2 rate too high = soft kill discipline)
  5. Identify bottlenecks using "5 Whys" pattern:
    • If R3 queue backing up: Why? ‚Üí Check R3 lead capacity ‚Üí Check R2 output quality ‚Üí Check R1 sourcing volume
    • If P1 tests delayed: Why? ‚Üí Check creative handoff ‚Üí Check store setup ‚Üí Check media buyer capacity
    • If test bank burning too fast: Why? ‚Üí Check P1 budgets ‚Üí Check kill discipline ‚Üí Check tier mix
  6. Classify issues by urgency:
    • RED - Critical: Blocks testing velocity (must resolve today)
    • YELLOW - Important: Friction building, will block soon (resolve this week)
    • GREEN - Monitor: Within tolerance, watch for trend (next review)
  7. Create action items for RED/YELLOW issues:
    • For RED: create ClickUp ops task with today's due date
    • For YELLOW: create ClickUp ops task with this week's due date
    • Assign to self or delegate to PM/pod lead as appropriate
    • Post in Discord if cross-pod coordination needed
  8. Document findings in daily health log:
    • Open Notion ‚Üí System Health Logs DB
    • Create daily entry with date, pod statuses, issues flagged, actions taken
    • Link to any created ClickUp tasks
  9. Brief Founder if critical issues:
    • Draft summary in Discord DM or #founder-brief channel
    • Include: issue, impact, proposed solution, decision needed (if any)
    • Don't wait for scheduled sync if RED issue exists

Outputs

  • Prioritized list of issues (RED/YELLOW/GREEN)
  • ClickUp ops tasks for RED and YELLOW items
  • Daily health log entry in Notion
  • Discord posts for cross-pod issues
  • Founder briefing if critical issues

Quality Check

  • ‚úì All WIP violations identified
  • ‚úì Test bank burn rate within bounds or flagged
  • ‚úì Critical blockers have same-day action items
  • ‚úì Health log is complete and searchable

Time Budget

30-60 minutes daily (more if critical issues require deep dive)

Qualitative Standards

Good vs. Bad: Canon Maintenance

‚úì Good Example ‚úó Bad Example
  • Canon change requires L1 update
  • Integrator drafts L1 v4.0 ‚Üí v4.1 change
  • Gets Founder approval
  • Updates Canon Registry (v4.0 = LEGACY, v4.1 = ACTIVE)
  • Regenerates 3 affected L4 SOPs
  • Announces change with links to new docs
  • Result: System stays coherent, all docs aligned
  • Pod requests canon change
  • Integrator verbally agrees but doesn't update docs
  • Change implemented in ClickUp only
  • Canon Registry not updated
  • SOPs still reference old rules
  • Result: Drift created, pods confused, new hires onboard to wrong canon

Good vs. Bad: Bottleneck Resolution

‚úì Good Example ‚úó Bad Example
  • Morning WIP check reveals R3 queue backing up
  • Integrator digs: R2 output quality is low, causing R3 rejections
  • Root cause: R1 volume too high, researchers rushing
  • Solution: Temporarily reduce R1 intake, add quality checkpoint
  • Documents in decision log, updates affected researchers
  • Monitors for 1 week, R3 queue normalizes
  • R3 queue backing up
  • Integrator tells R3 lead to "work faster"
  • No root cause analysis
  • R3 lead burns out trying to clear backlog
  • Quality drops further, more rejections
  • Bottleneck gets worse, testing velocity drops
üí° Leading vs. Lagging Indicators

Leading indicators (act on these early): WIP trending upward, R3 approval time increasing, escalation volume rising

Lagging indicators (outcome measures): Portfolio ROAS, testing velocity, winner rate

The Integrator's job is to spot and fix leading indicators before they degrade lagging indicators.
