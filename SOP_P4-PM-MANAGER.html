üéØ Role Purpose & Context

Why this role exists: To translate the CaniCol v4.0 model's strategic intent into daily operational reality across four specialized pods, ensuring the 45-60 products/month testing volume flows smoothly without bottlenecks or WIP overload.

Strategic Context

DriveOn Commerce operates under the CaniCol v4.0 lean-volume testing model. This model is built on systematic testing constraints:

  • Testing Volume: ~45-60 products/month reach P1/P2 stages
  • Economics: $40-60 per P1 test, $70-120 total P1+P2 budget per product
  • Hit Rate: Only 4-6% become true winners (~2-4 products/month)
  • Portfolio Distribution: 70-80% killed in P1, 10-16% reach P2, 4-10% reach P3

These constraints aren't arbitrary‚Äîthey're the mathematical backbone of a sustainable dropshipping operation. As PM Manager, your job is to protect these constraints while maximizing flow.

Why This Role Matters

Without dedicated project management, DriveOn would face:

  • Pod Siloing: P1 (Product Intelligence) doesn't know P2's (Creative) capacity; P2 doesn't know P3's (Media & Store) schedule
  • WIP Creep: Pods take on too much work, quality drops, throughput collapses
  • Invisible Blockers: Critical issues go unreported for days, cascading into missed deadlines
  • Chaos at Scale: What works at 20 products/month breaks at 50+

You exist to prevent these failure modes while keeping the system humane and sustainable for operators.

How You Support Holdings (L0) and Model (L1)

Holdings Level (L0): DriveOn Commerce is one product under DriveOn Holdings. Your coordination ensures Commerce delivers on its mandate: profitable, scalable product testing that generates 2-4 winners/month.

Model Level (L1): CaniCol v4.0 requires tight orchestration across three core phases (research, creative, testing) and four pods. You make the model real by:

  • Enforcing gate transitions (R3‚ÜíP1‚ÜíP2‚ÜíP3)
  • Protecting economic constraints (budgets, timelines)
  • Maintaining throughput without quality collapse

‚öôÔ∏è Core Workflows

These are your repeatable processes. Each workflow has a clear trigger, defined steps, and specific outputs in ClickUp/Notion/Discord.

WF1: Daily Blocker Review & Resolution

Trigger: Every morning (08:00-09:00) or when "#blocker" is mentioned in Discord

Inputs:

  • Discord #blockers channel messages
  • ClickUp tasks with Status = "Blocked"
  • Direct messages from pod members

Steps:

  1. Open ClickUp ‚Üí Switch to "Everything" workspace ‚Üí Open "Blocked Tasks" view (filter: Status = Blocked, all spaces)
  2. For each blocked task, read the task description and latest comment to understand the issue
  3. Categorize blocker:
    • Type A (Quick Fix): Missing info, simple question ‚Üí ping the blocker in Discord or task comments, set 2hr deadline
    • Type B (Coordination): Handoff stuck between pods ‚Üí schedule 15min call with both parties today
    • Type C (Resource/Capacity): Role at WIP cap or missing resource ‚Üí escalate to Integrator immediately
    • Type D (Canon/Process): Unclear process or canon conflict ‚Üí check canon docs, clarify in Discord, update SOP if needed
  4. Add a comment to the ClickUp task documenting your action and expected resolution time
  5. Update task status to "Active" only after blocker is resolved; otherwise leave as "Blocked" with clear next action
  6. Log blocker in your daily tracking sheet (Notion or ClickUp list): Date, Task ID, Blocker type, Resolution time, Outcome
  7. Post summary to #blockers channel: "Today's blockers: 3 resolved (avg 2hr), 1 escalated to Integrator, 0 pending"

Outputs:

  • ClickUp tasks unblocked or escalated
  • Discord summary posted to #blockers
  • Blocker log updated in Notion

Quality Check: No blocker should remain unaddressed for >4 hours during working hours. If you can't resolve, escalate.

Timing: 30-60 minutes/day (morning block)

WF2: Sprint Planning (Weekly Cadence)

Trigger: Every Monday 10:00 AM (or agreed sprint start day)

Inputs:

  • Last sprint's completed/incomplete tasks
  • Product pipeline forecast (how many R3 packets expected this week?)
  • Pod capacity data (who's at WIP cap, who's available?)
  • Integrator's strategic priorities for the week

Steps:

  1. Pre-Meeting Prep (1 day before):
    • Review last sprint: What % of committed tasks were Done? What blocked? Document in Notion sprint retro page
    • Pull current R3 packet count from ClickUp P1 space ‚Üí R3 Approved Packets list
    • Check P2 Creative capacity: How many hero edits are in progress vs capacity?
    • Check P3 Media buyer schedule: How many P1 slots available this week?
    • Draft proposed sprint goal (e.g., "Launch 12 P1 tests, complete 5 R3 packets, ship 3 brand store updates")
  2. Meeting (90-120 min):
    • Invite: Integrator, P1 Lead (or senior researcher), P2 Creative Director, P3 Media Buyer Lead, P3 Store Lead (if needed)
    • Agenda:
      1. Review last sprint (10 min)
      2. Discuss blockers and lessons (15 min)
      3. Present this week's pipeline forecast (10 min)
      4. Propose sprint goal and commitments (20 min)
      5. Assign specific tasks to roles/people (30 min)
      6. Flag any WIP or capacity concerns (10 min)
      7. Confirm handoff points and dependencies (10 min)
    • Use ClickUp screen share to move tasks from Backlog ‚Üí sprint board
    • Set due dates for all tasks assigned this sprint
  3. Post-Meeting:
    • Post sprint summary to #p4-pm-coordination: "Sprint X goals: [goals], commitments: [commitments], watch items: [risks]"
    • Update Notion Sprint Log page with sprint goal, committed tasks, key dependencies
    • Send individual reminders to task owners via ClickUp mentions or Discord DMs

Outputs:

  • ClickUp sprint board populated with tasks, due dates, assignees
  • Notion Sprint Log page updated
  • Discord sprint summary posted

Quality Check: Every task on sprint board must have: assignee, due date, Product_ID (if applicable), clear acceptance criteria in description

ü§ñ AI Usage & Prompt Patterns

AI (ChatGPT, Claude, etc.) can accelerate your PM work, but must never replace your judgment on canon compliance, capacity, or escalation decisions. Use AI for drafting, analysis, and pattern recognition‚Äînot for autonomous decision-making.

Where AI Helps

  • Sprint Planning Prep: Analyze last sprint's data, draft proposed commitments
  • Blocker Analysis: Identify patterns in recurring blockers, suggest root causes
  • Communication Drafting: Write Discord announcements, sprint summaries, escalation messages
  • Retrospective Synthesis: Summarize meeting notes, extract action items
  • Capacity Forecasting: Analyze throughput trends, predict bottlenecks

Where AI Does NOT Replace You

  • Canon Interpretation: AI doesn't have L0-L3 docs; you must verify all suggestions against canon
  • Escalation Decisions: AI can't judge when to escalate to Integrator; this requires human context
  • Conflict Resolution: Inter-pod conflicts need human mediation, not AI-generated "solutions"
  • WIP Judgment Calls: When to allow temporary WIP cap violation requires understanding team stress, not just rules

‚¨ÜÔ∏è Escalation & Decision Rights

Clear decision authority prevents both bottlenecks and overreach. Know what you can decide alone, what needs Integrator, and what needs Founder.

Decisions You Make Independently (No Approval Needed)

  • Daily Coordination: Moving tasks between statuses, reassigning within same role type, setting sprint due dates
  • Minor Schedule Adjustments: Shifting a handoff by 1-2 days if both pods agree
  • Blocker Resolution (Type A/B): Resolving info gaps, coordination issues, communication failures
  • Documentation Updates: Updating Notion sprint logs, blocker logs, weekly tracking sheets
  • Standard Reminders: Pinging team members about due dates, missing statuses, handoff delays
  • ClickUp Hygiene: Archiving old completed tasks, cleaning up duplicate tasks, updating descriptions for clarity

Decisions Requiring Integrator Approval

  • WIP Cap Exceptions: Allowing a role to exceed canon WIP cap, even temporarily
  • Capacity Changes: Recommending adding/removing FTE, hiring new roles, redistributing major responsibilities
  • Process Changes: Proposing changes to handoff protocols, status workflows, or sprint cadence
  • Systemic Blockers (Type C): Resource/capacity issues that can't be resolved by coordination alone
  • Schedule Conflicts: If two pods are deadlocked on priorities or timelines
  • Canon Clarifications: When there's ambiguity in L2-L3 docs and you need authoritative interpretation

Decisions Requiring Founder Approval

  • Strategic Changes: Proposing changes to testing volume (moving from 45-60 to different range)
  • Economic Changes: Recommending changes to P1/P2 budgets beyond canon ranges
  • Major Tool Changes: Proposing to add a new core tool to the stack (would trigger Canon Change Request)
  • Canon Conflicts: If you find a contradiction between L1 and L2 docs that Integrator can't resolve

Escalation Protocol

Standard Escalation Flow

  1. Document the Issue: Create ClickUp task or Notion page with:
    • Clear problem statement
    • Impact (which pods, which products, how many people affected)
    • What you've tried (so far)
    • Proposed solutions (2-3 options with tradeoffs)
    • Urgency (how soon needs decision)
  2. Send to Integrator: Discord DM + link to doc:
    • "@Integrator, need escalation on [issue]. Doc: [link]. Urgency: [timeline]. Request: [specific decision needed]"
  3. If Urgent (needs decision <24hrs): Tag in relevant channel too:
    • "@Integrator, urgent escalation in DM re: [issue], affects [pod/deadline]"
  4. Follow Up: If no response in expected timeframe (24hrs non-urgent, 4hrs urgent), send reminder:
    • "Following up on [issue] escalation from [date]. Still need decision by [deadline]."
  5. Document Decision: Once Integrator decides, update original doc and post summary to relevant channels:
    • "Update on [issue]: Decision made, [action]. Details: [link]"

Escalation Type Expected Response If No Response
