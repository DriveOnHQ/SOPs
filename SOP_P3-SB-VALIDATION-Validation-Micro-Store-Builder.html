2. Role Purpose & Context
Why This Role Exists in CaniCol v4.0

Strategic Purpose

The Validation / Micro Store Builder is the rapid deployment engine for DriveOn Commerce's CaniCol v4.0 testing model. This role converts winning creative concepts (P2 outputs) into conversion-optimized validation stores that enable fast, cost-effective product-market fit testing within the tight P3 budget constraints.

Economic Context

CaniCol v4.0 operates on lean-volume portfolio economics:

  • ~45-60 products/month enter P1/P2 testing
  • P1+P2 budget: $70-120/product (standard band $80-100)
  • Hit rate: 4-6% (2-4 true winners per month)
  • Portfolio thinking: rapid kill, fast scale winners

This role enables that model by:

  1. Speed: Deploying stores in 3-4 hours (not days), keeping P3 validation cycles tight
  2. Consistency: Using templated Shrine theme + proven page builder patterns to reduce variance
  3. Conversion focus: Building stores that convert at ‚â•1.5% baseline, maximizing signal from P3 ad spend
  4. Scalability: Handling 8-12 stores/week per FTE without founder heroics

Pod 3 Integration

Pod 3 (Media & Store) is responsible for validation, scaling, and operations. The Store Builder role sits at the critical handoff point between:

  • Upstream (P2 Creative): Receives creative assets, product briefs, angle decks
  • Downstream (P3 Media Buying): Delivers launch-ready stores with tracking, compliance, and conversion optimization

Without this role, P3 validation would be bottlenecked by slow, inconsistent store builds or forced into low-quality "quick launch" stores that muddy test data.

Holdings Alignment

Per L0-02 Product & Brand Architecture, DriveOn Commerce is a dropshipping engine brand focused on high-velocity testing. The Store Builder role directly enables the "engine" metaphor by treating store deployment as a repeatable, standardized manufacturing process, not a creative craft project.

      • Detailed Description: Long-form copy from brief
      • FAQs: 3-5 common questions (use template if product brief doesn't have specific FAQs)
      • Trust Signals: Guarantee, shipping, secure checkout badges

  3. 3 Mobile Optimization Pass:
    • Switch page builder to mobile preview mode
    • Check all sections render correctly on mobile
    • Adjust:
      • Font sizes (headlines ‚â•24px, body ‚â•16px)
      • Button sizes (‚â•44px height for tap targets)

  • Copy Quality: No typos, matches angle deck, benefit statements are clear and specific (not vague)
  • Mobile Experience: Text is readable, buttons are tappable, no horizontal scroll, layout doesn't break

8. AI Usage & Prompt Patterns
Where and How This Role Uses AI

The Store Builder role leverages AI for speed and consistency, not to replace human judgment. AI is used for ideation, draft generation, and QA assistance‚Äîalways with human review before implementation.

AI Usage Map

1. Copy Optimization & Refinement

Use Case: When P2 creative copy needs to be adapted for store context (e.g., homepage vs product page) or when copy feels weak/generic.

Prompt Pattern:

You are a direct-response copywriter for e-commerce product pages.

Product: [Product Name]
Primary Angle: [From angle deck]
Target Audience: [From product brief]

Current Copy:
[Paste P2 copy here]

Task: Rewrite this copy for a product page "benefits" section. Make it:
1. More specific (avoid vague words like "amazing" or "great")
2. Benefit-focused (not feature-focused)
3. Scannable (3-5 short bullet points)
4. Action-oriented (use active verbs)

Output format: Bullet list, each bullet ‚â§15 words.

Human Checkpoint: Review AI output for tone, accuracy, and compliance. Edit before publishing.

2. Page Layout & Element Suggestions

Use Case: When stuck on homepage structure or unsure which elements to prioritize for a specific product type.

Prompt Pattern:

I'm building a Shopify store for a [product category] product.

Product Details:
- Product: [Product Name]
- Price Point: $[X]
- Target Audience: [Audience]
- Primary Benefit: [Main selling point]

I'm using [GemPages/PageFly/Replo] and have a 5-section homepage template.

Question: What are the 5 most important homepage sections for this product type, in priority order?

For each section, suggest:
1. Section name
2. What to include (be specific, not generic)
3. Why it matters for this product type

Human Checkpoint: Validate suggestions against product brief and angle deck. Adapt to fit creative assets available.

3. QA Checklist Generation

Use Case: For new product categories or complex stores, generate a custom QA checklist supplement.

Prompt Pattern:

I'm QA-testing a Shopify store for a [product category] product (e.g., health supplement, pet product, home decor).

I have a standard QA checklist that covers:
- Functional tests (links, cart, checkout)
- Performance tests (load time, mobile)
- Content tests (typos, copy accuracy)
- Compliance tests (legal pages, no false claims)

Task: Generate 5-10 additional QA checks specific to [product category] that I should add.

For each check, provide:
1. What to test
2. How to test it (specific steps)
3. Pass/fail criteria

Human Checkpoint: Review for relevance and feasibility. Add to standard checklist if useful.

4. CRO Hypothesis Generation

Use Case: When optimizing a live store (Workflow 5) and need ideas for A/B tests or improvements.

Prompt Pattern:

I'm optimizing a live Shopify store with these performance metrics:

- Sessions (7 days): [X]
- Conversion Rate: [Y%]
- Bounce Rate: [Z%]
- Cart Abandonment: [A%]
- Avg Session Duration: [B seconds]

Product: [Product Name]
Primary Issue: [e.g., "high bounce rate", "low ATC rate", "high cart abandonment"]

Task: Suggest 3 CRO optimizations to test, prioritized by potential impact.

For each optimization, provide:
1. What to change (be specific, e.g., "Replace hero image with lifestyle video")
2. Hypothesis (why this will improve metrics)
3. Implementation difficulty (Easy/Medium/Hard)
4. Expected impact (Low/Medium/High)

Human Checkpoint: Validate suggestions with data. Choose highest-impact, lowest-effort options. Test one at a time.

AI Usage Rules

  • NEVER publish AI-generated content without human review: All copy, layout suggestions, and QA checks must be reviewed by Store Builder before implementation
  • NEVER use AI for compliance decisions: AI cannot determine if content violates L2-09 Offer Ladder & Risk Guardrails. Human judgment required.
  • NEVER let AI change product details: AI should not modify product pricing, specs, or supplier info. These come from P1 research packets and are non-negotiable.
  • ALWAYS cross-check AI suggestions against canon: If AI suggests tools, processes, or structures not in L2/L3 canon, flag as non-canon and do not implement.

Recommended AI Tools

  • ChatGPT (GPT-4): Copy refinement, layout suggestions, QA checklists
  • Claude (this tool): Complex reasoning, multi-step workflows, canon alignment checks
  • Grammarly: Spell check and grammar QA (not strictly AI, but useful)

Note: Do not use AI image generators for product images. All images come from P2 Creative or supplier. AI-generated images are not acceptable for validation stores.

10. Escalation & Decision Rights
Autonomy, Approval Requirements, and When to Escalate

Decisions the Store Builder Can Make Alone (No Approval Needed)

  • Which page builder to use for a specific product (GemPages, PageFly, or Replo)
  • Specific layout and design choices within a store (section order, image placement, etc.)
  • Minor copy edits to improve clarity or fix typos (as long as core angle/message is preserved)
  • When to deploy standard CRO optimizations (if following best practices and no major changes)
  • Daily priority order for store builds (which stores to work on first)
  • Whether to accept or reject a P2 handoff based on completeness criteria (Workflow 1, step 1)

Decisions Requiring Pod Lead or PM Approval

  • Changing core product positioning or angle: If P2 creative angle feels wrong or needs major revision ‚Üí escalate to P3 Pod Lead + P2-CR-LEAD
  • Adding custom code or advanced features: Anything beyond standard page builder capabilities (e.g., custom Liquid code, third-party integrations) ‚Üí requires PM approval
  • Exceeding WIP limits: If you need to take on >4 active builds or >8 optimization tasks ‚Üí requires P3 Pod Lead approval
  • Major compliance gray areas: If unsure whether product or copy violates L2-09 ‚Üí escalate to PM or Legal (if available)
  • Process changes to workflows: If you want to modify standard workflows (e.g., skip QA for certain products) ‚Üí requires PM approval + Canon Change Request

When to Escalate to Founder / Canon Owner

  • Canon conflicts or contradictions: If you discover a conflict between L0/L1/L2/L3 docs that affects store builds ‚Üí escalate to Canon Owner (via PM)
  • Major tool or platform issues: If Shopify, page builders, or core tools are broken and blocking multiple stores ‚Üí escalate to Founder (via P4-OP-PM)
  • Economic or strategic questions: If you're unsure whether a product fits the CaniCol model (e.g., very high AOV, unusual supplier terms) ‚Üí escalate to Founder
  • Ethical or legal concerns: If a product or offer feels unethical, illegal, or reputationally risky ‚Üí escalate immediately to Founder (do not proceed with build)
